# 🧪 데모 리허설 및 테스트 플랜 (Demo Test Plan)

성공적인 팀 데모를 위해 **3단계(Phase)**로 나누어 검증 및 수정을 진행합니다.

---

## 📅 전체 일정 요약

| 단계 | 목표 | 대상 시나리오 | 예상 소요 |
|------|------|---------------|-----------|
| **Phase 1** | **기능 단위 테스트** | 단순 앱 실행, CLI/Shell 명령 | 10분 |
| **Phase 2** | **복합 시나리오 완주** | 모닝 브리핑, 지출 결의서 등 | 30분 |
| **Phase 3** | **예외 처리 및 최적화** | 오류 발생 시 복구(Self-Healing) 검증 | 20분 |

---

## ✅ Phase 1: 기능 단위 테스트 (Sanity Check)
> **목표:** 개별 모듈이 정상 작동하는지 확인 (이거 안 되면 데모 불가).

1. **Vision Check**: `surf 계산기` -> 계산기가 열리는가? (Gemini CLI 연동 확인)
2. **Shell Check**: `surf ls -la` -> 현재 디렉토리 파일 목록이 보이는가? (Executor 확인)
3. **App Check**: `surf 사파리` -> 브라우저가 뜨는가? (MacOS 제어 확인)

**🔴 실패 시 조치:**
- `.env` 키 값 재확인.
- `steer.db` 삭제 후 재시작 (DB 락 해제).

---

## 🚀 Phase 2: 복합 시나리오 드라이런 (Dry Run)
> **목표:** 4대 시나리오가 끊김 없이(End-to-End) 실행되는지 검증.

1. **🌅 모닝 브리핑 (Gmail -> Notion)**
   - 메일 읽어오는지 확인.
   - Notion 페이지에 텍스트가 추가되는지 확인.
2. **💰 지출 결의서 (Notion -> Excel)**
   - 엑셀 셀에 숫자가 정확히 입력되는지 확인 (오타율 체크).
3. **🔍 마켓 리서치 (Web -> Slack)**
   - 검색 결과 캡처 이미지가 깨지지 않는지 확인.
4. **🚨 장애 대응 (Shell -> Jira)**
   - 로그 파일(`error.log`) 없으면 가짜 파일 생성해두기.

**🔴 실패 시 조치:**
- **타이밍 이슈:** `visual_driver.rs`의 `wait_for_ui_settle` 시간(기본 2초)을 3~4초로 늘림.
- **좌표 이슈:** 해상도 스케일링(Retina) 설정 확인.

---

## 🛡️ Phase 3: 예외 처리 및 최종 최적화 (Robustness)
> **목표:** 데모 중 에러가 터져도 자연스럽게 수습하는지 확인.

1. **인터럽트 테스트:** 앱 실행 중 팝업창(예: 업데이트 알림) 띄워보기 -> **ESC로 닫는지 확인.**
2. **명령어 모호성:** "그거 해줘" 같이 애매하게 말했을 때 -> **"명확히 말씀해주세요"라고 되묻는지 확인.**
3. **최종 리허설:** 시나리오 1~4를 연속으로 실행하며 배터리/네트워크 상태 점검.
