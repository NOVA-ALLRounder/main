# AI Agents Lifecycle (Smart Folder Scope)

### AI-summary 프로젝트 전용 · Full Multi-View Technical Spec

*(Serverless · Free Models · Local Execution)*

---

## 0. 문서 목적 및 범위

이 문서는 **AI-summary 프로젝트**에서 동작하는 AI 에이전트들의 전체 생명주기(Lifecycle)를 **Cycle 0~5**로 정의하고, 각 Cycle을 **4가지 관점(PM / Tech Lead / UX / ML)**에서 설명한다.

핵심 목표:

* Smart Folder 기반 **데이터 경계**를 명확히 정의한다.
* `scan → train → chat → watch`로 이어지는 **파이프라인의 단계별 역할과 책임(R&R)**을 구조적으로 정리한다.
* 서버/클라우드 없이 **로컬 무료 모델(Whisper.cpp, Allama, MiniLM 등)**만 사용하는 환경에서, 현실적으로 구현 가능한 **실행 기준**을 제공한다.
* 개발자, 기획자, UX, ML 담당자가 **같은 문서를 보고도 각자 필요한 깊이로 해석**할 수 있게 한다.

이 문서는 **API 스펙이나 코드 문서가 아니라**,
AI-summary의 **엔진/에이전트 작동 방식에 대한 “공식 운영 설계서”**에 해당한다.

---

## 1. 공통 전제 (Global Assumptions)

### 1.1 Smart Folder & Policy

* **Smart Folder**:

  * 사용자 또는 운영자가 “이 안에 있는 자료만 AI가 사용해도 된다”고 명시적으로 허용한 폴더 집합.
* `smart_folders.json`(현재 레포: `core/config/smart_folders.json`)에 각 폴더의 `scope`, `base_path`, `policyPath`, `allowed_types` 등이 기록된다.
* **Policy 파일** (`policyPath`):

  * 허용 확장자, 허용/차단 경로 패턴, 권한 요구 사항 등을 정의.
  * 정책 위반 시, 해당 파일은 **즉시 제외**되며 파이프라인은 Fail-Closed를 기본으로 한다.

### 1.2 서버리스 & 무료 모델 환경

* 서버/클라우드 인프라 없음 →
  **모든 처리(STT, 임베딩, LLM 추론, 색인)는 로컬 단일 머신에서 실행**된다.
* 유료 API 호출 없음 →
  **Whisper.cpp, Allama, MiniLM/Instructor 임베딩 등** 무료/로컬 가능한 모델과 라이브러리만 사용한다.
* 데이터/캐시/로그는 모두 **로컬 파일 시스템**에 저장:

  * `data/` – corpus, parquet, joblib 등의 아티팩트
  * `data/cache/` – 검색 인덱스/임베딩 캐시 (예: `doc_meta.json`, `doc_embeddings.npy`, `doc_index.faiss`, `translations.sqlite3`)
  * `artifacts/logs/` – 런타임/감사/audit, drift/resource 로그 등

### 1.3 Lifecycle 공통 구조

모든 에이전트(미팅/지식/추가 역할)는 아래 Cycle 구조를 공유한다.

> **Cycle 0. 온보딩 (Onboarding)**
> **Cycle 1. 데이터 수집 (Scan)**
> **Cycle 2. 실행 (Processing / Agent Work)**
> **Cycle 3. 결과 제공 (Output Delivery)**
> **Cycle 4. 감사 & 피드백 (Audit & Feedback)**
> **Cycle 5. 재학습 / 최적화 (Retrain / Optimize)**

그리고 이들을 가로지르는 **공통 지원 사이클(Common Support)**가 있다.

## 2. Cycle 0. 온보딩 (Installation & Provisioning)

### 2.1 PM 관점

**목적**

* “AI-summary가 **어디까지를 쓸 수 있고, 어디부터는 절대 건드리면 안 되는지**”를 처음에 확정하는 단계.
* 사용자의 기대와 보안·정책 요구사항을 동시에 만족하는 초기 경계를 설정.

**해야 할 일**

1. **스마트 폴더 선택**

   * 사용자가 하나 이상의 폴더를 선택.
   * 하위 디렉터리 포함 여부, 특정 패턴 제외 여부 등 옵션 포함.

2. **정책 매핑**

   * 각 스마트 폴더에 대해 사용할 정책 파일(`policyPath`) 연결.
   * 예: 회의용 폴더는 오디오/문서 위주, HR 폴더는 PII/민감도 강화 정책 등.

3. **프리셋 제공**

   * “회의/요약 전용”, “문서 검색 전용” 같은 미리 정의된 프리셋.
   * PM 입장에서는 이 프리셋이 “제품 레벨 옵션”에 해당.

4. **결과 저장**

   * 최종 설정은 `smart_folders.json`에 영구 저장.
   * 이후 모든 Cycle은 이 파일에 정의된 scope 밖으로 나갈 수 없음.

**성공 기준**

* Smart Folder와 Policy가 **모두 유효한 경로/파일로 설정**된다.
* PM이 기대하는 **데이터 경계**가 기술적으로도 정확히 반영된다.

---

### 2.2 Tech Lead 관점

**입력**

* 사용자가 선택한 폴더 경로 목록
* 각 폴더에 대한 policy 파일 경로
* 허용 확장자/패턴 규칙

**처리**

1. **경로 검증**

   * 존재 여부, 권한 확인, 심볼릭 링크/네트워크 드라이브 여부 점검.
2. **Policy 파싱**

   * YAML/JSON 기반 정책 파일을 로딩해 내부 구조 검증.
3. **프리뷰 시뮬레이션 (선택)**

   * 몇 개 파일을 샘플링해서 “허용/차단” 여부를 미리 계산.
4. **구성 저장**

   * `smart_folders.json`에 구조화된 형태로 저장:

     * `scope_id`, `base_path`, `policyPath`, `allowed_types`, `recursive` 등.

**출력**

* 유효한 Smart Folder 설정이 반영된 `smart_folders.json`
* 초기 로그: 온보딩이 언제/어떤 설정으로 수행되었는지

**비기능 요구사항**

* 온보딩 실패 시 **이후 모든 파이프라인 호출은 즉시 중단**해야 한다.
* Smart Folder 설정은 나중에 UI에서 재실행 가능해야 한다(리온보딩).

---

### 2.3 UX 관점

* 온보딩 UI는 **복잡한 설정을 최대한 숨기고**,
  “**이 폴더 안 자료만 AI가 본다**”는 메시지를 강조해야 한다.
* 정책 오류/권한 문제 발생 시:

  * “접근 권한이 없습니다.”가 아니라
  * **“이 폴더는 OS 권한이 없어 읽을 수 없어요. 관리자 권한으로 다시 실행하거나 다른 폴더를 선택해 주세요.”** 같은 해결책 포함 메시지.
* 사용자가 **어떤 데이터가 쓰이고, 어떤 데이터는 절대 안 쓰이는지**를
  프리뷰(허용/차단 샘플 리스트)로 한 번은 눈으로 확인하게 해주는 것이 중요.

---

### 2.4 ML 관점

* 이 단계에서는 실제 모델 추론은 수행하지 않지만,

  * **추후 STT/LLM/임베딩 모델이 처리할 데이터 양과 유형을 가늠**할 수 있는 정보가 모인다.
* ML 관점에서 중요한 포인트:

  * 오디오 파일 비중이 크면 STT 리소스 플랜 필요.
  * PDF/문서 비중이 크면 텍스트 추출/전처리 비용 고려 필요.
* 따라서 온보딩 시 **파일 타입 통계**를 간략히 수집해두면 이후 ML 전략(Chunking, Batch 크기 등)에도 도움된다.

---

## 3. Cycle 1. 데이터 수집 (Scan)

### 3.1 PM 관점

**목적**

* Smart Folder 내 파일들을 “AI가 다룰 수 있는 후보 리스트”로 만드는 단계.
* 실제 “학습/색인”에 들어가기 전, **데이터 인벤토리**를 확립하는 단계.

**비즈니스적 의미**

* 어떤 문서/오디오가 실제로 분석 대상인지 투명하게 보여줄 수 있어야 한다.
* “왜 이 문서는 검색 안 되지?”에 대한 1차 답은 모두 **Scan 단계 결과**로 설명 가능해야 한다.

---

### 3.2 Tech Lead 관점

**입력**

* `smart_folders.json`
* 정책 파일들 (`policyPath`)

**처리**

1. **폴더 트래버설**

   * 각 Smart Folder별로 재귀적으로 파일 탐색.
   * 정책에 기반해 허용 확장자/패턴 필터링.

2. **메타데이터 추출**

   * 파일 경로, 크기, 수정 시간, MIME 타입 등의 최소 메타데이터 수집.
   * 결과를 `found_files.csv` 같은 형태로 저장.

3. **차단 파일 처리**

   * 허용되지 않는 타입/경로는 스캔 로그에 “차단” 사유와 함께 기록.
   * 이 정보를 UX에서 시각화할 수 있도록 구조화.

4. **Watch 모드와 연계 (선택)**

   * `watch` 모드 실행 시 파일 시스템 변경 이벤트(inotify 등)를 이용해
     변경/신규 파일을 감지하고, Scan 로직을 증분으로 재사용.

**출력**

* 스캔 결과 CSV/JSON (허용 파일 리스트)
* 차단 파일 리스트(간단한 사유 포함)
* 선택적으로 파일 타입/폴더별 통계

---

### 3.3 UX 관점

* Scan 결과 화면에서 사용자에게 다음을 보여줘야 한다:

  * “총 N개 파일 중, M개가 분석 후보로 선정, K개는 정책상 제외되었습니다.”
  * 어떤 타입이 많이 포함되어 있는지(오디오/문서/기타).
* 차단된 파일에 대해서:

  * **“이유를 한 줄로”** 보여줘야 함.

    * 예: “허용되지 않은 확장자(.xlsx)”, “정책에서 제외된 하위 폴더 경로”.
  * 디버깅용 내부 정보(정책 키 이름 등)는 숨기고, Plain Language 위주로.

사용자 경험 측면에서 이 단계는:

> “지금부터 이 폴더 안에 있는 이 파일들을 가지고 AI가 일할 거야.”
> 라는 **신뢰 형성 단계**이다.

---

### 3.4 ML 관점

* Scan 단계는 ML 추론을 하지 않지만,

  * **코퍼스 구성의 1차 필터** 역할을 한다.
* ML에서 중요하게 보는 포인트:

  * 문서/오디오의 개수와 총 용량.
  * 파일 타입별 비율 (텍스트 vs PDF vs 오디오 등).
* 이 정보는:

  * **Chunking 전략**(텍스트 당 토큰 수, 문서 분할 방식),
  * **임베딩 리소스 추정**(시간, 메모리),
  * **STT GPU/CPU 리소스 플랜**을 잡는 데 활용된다.

## 4. Cycle 2. 실행 (Processing / Agent Work)

Cycle 2는 크게 두 개의 에이전트 작업으로 나뉜다.

1. **Meeting Agent**: 오디오/회의 → STT → 요약 → 액션 아이템
2. **Knowledge Agent**: 문서 → 전처리 → 임베딩/색인 → 질의응답(RAG)

---

### 4.1 Meeting Agent

#### 4.1.1 PM 관점

**목적**

* 회의 녹음 파일을 받아,
  **STT → 요약 → 하이라이트 → 액션 아이템**까지 한 번에 생성.
* 사용자는 “녹음 파일만 집어넣으면, 회의록+To-do 리스트까지 다 나오는” 경험을 얻는다.

**핵심 요구사항**

* 스마트 폴더 경계 밖 자료 사용 금지.
* 액션 아이템에 **Owner/기한/우선순위** 같은 최소 메타데이터 포함.
* 내부/외부 공유 시 **마스킹 정책** 차등 적용.

---

#### 4.1.2 Tech Lead 관점

**입력**

* STT 대상 오디오 파일 경로 (Scan 결과 기반)
* Smart Folder Context (scope, policyPath)

**처리 플로우**

1. **오디오 Chunking**

   * 긴 파일을 5분 내외 단위로 분할.
   * Whisper.cpp 호출 시 메모리/성능 안정성을 위해 필요.

2. **STT (Whisper.cpp)**

   * 각 Chunk에 대해 STT 수행.
   * 결과를 하나의 텍스트로 병합, 타임코드 유지.

3. **요약 & 하이라이트 생성**

   * 전체 텍스트 또는 Chunk 단위 요약 수행.
   * LLM(Allama 기반 LLaMA/Mistral) 사용.
   * 요약 결과를 구조화:

     * 전체 요약
     * 주요 논점 리스트
     * 중요한 결정 사항

4. **액션 아이템 추출**

   * 규칙 기반(“~해야 한다”, “~하기로 했다” 등 패턴) + LLM 보조.
   * 각 아이템에:

     * 설명
     * 담당자 후보(이름/역할)
     * 기한(있다면)
     * 중요도(High/Medium/Low)

5. **정책/마스킹 적용**

   * 내부 뷰: “홍*동”, “이*영” 같은 부분 마스킹.
   * 외부 공유용: 이름, 메일, 전화번호 등 PII 전체 마스킹 혹은 제거.

**출력**

* 회의 요약/하이라이트/액션 아이템 JSON/Markdown
* Smart Folder 내부에 저장된 결과 파일

---

#### 4.1.3 UX 관점

* UI에서 보여줄 때:

  * **요약 / 하이라이트 / 액션 아이템**을 **한 화면에서** 카드형으로 정리.
  * 액션 카드에 담당자/기한을 시각적으로 배치.
* 사용자는:

  * “전체 요약 → 세부 하이라이트 → 액션” 순으로 자연스럽게 내려가며 읽을 수 있어야 함.
* STT 품질 낮을 때:

  * “음질이 좋지 않아 일부 인식이 틀릴 수 있습니다” 같은 안내 배너 필요.

---

#### 4.1.4 ML 관점

* STT:

  * Whisper.cpp 모델 사이즈 선택(예: small/base)을 환경에 맞게 튜닝.
  * 노이즈/에코가 심한 환경 대비, 후처리(필터링) 전략 필요.
* 요약:

  * Prompt 설계 중요:

    * 회의록 구조(Agenda → Decision → Action)를 강조하는 Prompt.
* 액션 추출:

  * 규칙 기반과 LLM 추출 결과를 비교해, **중복 제거/신뢰도 계산** 필요.
* 장기적으로:

  * 액션 아이템 피드백(“이건 액션이 아니다” 같은 사용자 피드백)을 활용해
    추출 규칙을 개선할 수 있음.

---

### 4.2 Knowledge Agent (Indexing & RAG QA)

#### 4.2.1 PM 관점

**목적**

* Smart Folder 내 문서를 기반으로,
  사용자가 질문을 던지면 해당 범위 안에서만 **근거를 바탕으로 답변**을 주는 지식 검색 에이전트를 만든다.

**비즈니스 요구사항**

* “이 시스템이 어디까지의 문서를 보고 답을 하는지”가 명확해야 함.
* 답변에는 항상 **근거 문서**가 표시되어 신뢰를 줄 수 있어야 한다.

---

#### 4.2.2 Tech Lead 관점

**색인(Indexing)**

* 입력: Scan 결과의 허용 문서 리스트.
* 처리:

  1. 텍스트 추출(PDF, docx 등).
  2. 문단/섹션 기준으로 Chunking (예: 512~1024 토큰).
  3. MiniLM/Instructor 임베딩 생성.
  4. 벡터 인덱스(FAISS/SQLite+JSON) 파일 생성.
* 출력:

  * 기본 캐시 디렉터리(예: `data/cache/`)에 인덱스/메타 저장 (`doc_index.faiss`, `doc_meta.json`, `doc_embeddings.npy`)

**검색(Retrieval & RAG)**

* 입력:

  * 사용자 질문
  * Smart Folder scope
* 처리:

  1. 질문을 동일 임베딩 모델로 변환.
  2. Top-K 유사 Chunk 검색.
  3. 검색된 Chunk들을 LLM 컨텍스트로 전달.
  4. RAG 답변 생성.
* 출력:

  * 자연어 답변 + 인용 정보(문서명, 섹션, snippet)

---

#### 4.2.3 UX 관점

* 검색 결과 화면에서:

  * 최상단에 답변 텍스트.
  * 아래에 **근거 문서 리스트** (제목, 섹션, 짧은 발췌).
* “자료 없음” 상황과 “정책 차단으로 인해 표시 불가” 상황을 구분:

  * 자료 없음: 회색 아이콘 + “해당 스마트 폴더 내에 관련 자료를 찾지 못했습니다.”
  * 정책 차단: 자물쇠 아이콘 + “현재 권한/정책으로는 이 자료를 표시할 수 없습니다.”
* 대용량 문서는 전체를 열지 않고:

  * 인용 구간 주변만 하이라이트해서 보여주는 뷰어 제공.

---

#### 4.2.4 ML 관점

* 임베딩 모델:

  * MiniLM/Instructor 계열 → 성능/속도 균형.
* Top-K:

  * 너무 크게 잡으면 LLM 컨텍스트가 지저분해지고 비용 증가.
  * 일반적으로 3~5 사이 값 사용.
* RAG 전략:

  * 단순히 문단을 붙여주는 게 아니라,
    “각 문서가 어떤 의미에서 이 답변의 근거인지”를 요약해서 함께 제공하는 Prompt 설계 고려.
* 언어:

  * 다국어 문서 혼재 시, 임베딩 모델이 다국어 지원되는지 확인 필요.

---

## 5. Cycle 3. 결과 제공 (Output Delivery)

### 5.1 PM 관점

* 사용자가 얻는 산출물은 크게 두 가지:

  1. **회의 요약/액션 결과** (Meeting Agent)
  2. **질의응답 + 근거 문서 묶음** (Knowledge Agent)
* 이 산출물은 모두 **Smart Folder 내부 파일로 저장**되며,
  외부 공유는 정책이 허용하는 범위의 “마스킹된 내보내기”로만 가능.

---

### 5.2 Tech Lead 관점

**출력 형태**

* 회의 결과:

  * `YYYY-MM-DD_meeting_summary.md` 등
  * JSON 구조 결과도 병행 저장 가능.
* 검색 결과:

  * 쿼리 로그 + 인용 문서 ID 목록 등.

**경계 관리**

* 어떤 경우에도 **Smart Folder 외부 경로로 자동 저장/Export 금지**.
* Export 시:

  * 사용자가 명시적으로 경로를 선택하되,
  * 민감 데이터 제거 여부를 다시 한 번 확인하는 절차가 필요(UX와 연동).

---

### 5.3 UX 관점

* 사용자가 결과 화면에서 할 수 있어야 하는 것:

  * 요약 읽기
  * 링크 클릭으로 근거 문서 일부 보기
  * 결과 텍스트 복사
  * PDF/Markdown 등으로 내보내기
* 중요한 점:

  * “결과” 자체뿐 아니라,
    **“이 결과가 어디서 나왔는지”**를 항상 같이 보여줘야 신뢰가 생긴다.

---

### 5.4 ML 관점

* 결과 제공 단계에서는 ML 추론보다:

  * **결과의 신뢰도와 한계**를 어떻게 표현할지가 중요.
* Confidence Score가 낮은 결과:

  * UI에서 경고 배지 표시 가능.
  * “참고용으로만 사용해 주세요” 같은 안내 문구.

---

## 6. Cycle 4. 감사 & 피드백 (Audit & Feedback)

### 6.1 PM 관점

* 이 단계는 시스템이 잘 동작하는지,

  * 정책 위반이 없는지,
  * 사용자가 만족하는지,
    를 **정량/정성 모두로 측정하는 단계**다.

* 목적:

  * “이 시스템을 계속 써도 되는가?”
  * “어디를 손봐야 하는가?”에 대한 근거 확보.

---

### 6.2 Tech Lead 관점

**로그 체계**

* `artifacts/logs/audit/YYYY-MM-DD.jsonl`:

  * 주요 이벤트 (Scan, Train, Query, Error, Policy violation 등)
  * 타임스탬프, scope, 파일 수, 에러 코드 등 기록.
* `feedback.jsonl`:

  * 사용자 피드백(👍/👎, 간단 코멘트 등) 기록.

**요구사항**

* 로그는 **Append-only** 성격.
* 에러 상세는 여기 기록되지만, 사용자에게는 Plain Message만 보여줌.

---

### 6.3 UX 관점

* 피드백 UI는 최대한 간단히:

  * 결과 화면 아래에 “도움이 되었나요? 👍 / 👎”
  * 👎 시, 선택적으로 간단 사유 입력(“정확하지 않음”, “문맥과 안 맞음” 등).
* 장애 상황:

  * “무언가 잘못되었습니다 (Error 503)” 같은 메시지는 금지.
  * 대신:

    * “일시적으로 회의 요약 기능을 사용할 수 없습니다.
      데이터는 안전하게 보호되고 있으며, 잠시 후 다시 시도해 주세요.”
  * 기술적 디테일은 로그로만.

---

### 6.4 ML 관점

* 피드백 데이터는 ML을 위한 gold label은 아니지만,

  * **문제 구간을 좁히는 힌트**로 쓸 수 있음:

    * 특정 타입 문서에서만 정확도가 떨어진다.
    * 특정 언어/도메인에서 성능 저하.
* 이를 기반으로:

  * Chunking 전략 조정,
  * 임베딩 모델 교체/튜닝,
  * 요약 Prompt 개선 등을 할 수 있다.

---

## 7. Cycle 5. 재학습 / 최적화 (Retrain / Optimization)

### 7.1 PM 관점

* 이 시스템은 **“자동 ML 재학습”을 하지 않는다.**
* 대신:

  * 데이터/정책이 변화할 때,
  * 새로운 문서/회의가 많이 쌓였을 때,
  * 품질 문제가 계속 리포트될 때,
    → **재스캔 & 재색인**으로 품질을 유지·개선한다.

---

### 7.2 Tech Lead 관점

* 재학습/최적화는 크게 세 종류:

1. **재스캔 & 재색인**

   * Scan → Train 단계 재실행.
   * corpus, data/cache 업데이트.

2. **모델 교체/튜닝**

   * Whisper 모델 사이즈 변경,
   * Allama 내부에서 사용하는 LLM 모델 변경,
   * 임베딩 모델 교체.

3. **설정 변화 반영**

   * Smart Folder 또는 Policy 변경 시,
   * 관련 scope는 전체 재처리가 필요할 수 있음.

---

### 7.3 UX 관점

* 사용자에게:

  * “현재 인덱스를 최신 상태로 재구성 중입니다.
    검색 결과가 일부 누락될 수 있습니다.” 같은 안내 필요.
* 재색인 작업은 긴 시간이 걸릴 수 있으므로:

  * 진행률 또는 대략적인 예상 시간 표시.

---

### 7.4 ML 관점

* 재학습보다는 **재색인·재전처리**에 초점을 맞춘다.
* Whisper/LLM/임베딩 모델 자체를 바꾸는 경우:

  * 기존 결과와 새 결과의 **비교 테스트**가 필요.
  * 예: 동일 쿼리/회의에 대한 요약/검색 결과 비교.

---

## 8. 공통 지원 사이클 (Common Support Cycle)

### 8.1 PM 관점

* 여러 에이전트가 함께 동작할 때,

  * Meeting Agent의 요약 결과가 Knowledge Agent의 검색 대상이 될 수 있다.
* 그러나 이때도 **항상 Smart Folder scope를 유지**해야 한다.

  * A 폴더의 회의 요약이 B 폴더의 검색에 몰래 쓰이면 안 된다.

---

### 8.2 Tech Lead 관점

* 모든 요청에는 항상 Context Object가 따라붙어야 한다:

  * `{ scope_id, policyPath, trace_id, allowed_types }`
* 에이전트 간 데이터 전달 시:

  * scope_id가 일치하지 않으면 바로 차단(Fail-Closed).
* 정책 변경/폴더 구조 변경이 감지되면:

  * 관련 scope에 한해 재온보딩/재스캔 트리거.

---

### 8.3 UX 관점

* 교차 에이전트 시나리오:

  * “이 회의에서 나온 액션 관련 문서를 찾기” 같은 기능.
* 이때:

  * “지금 보고 있는 회의와 같은 Smart Folder 안에서만 문서를 찾습니다”
    라는 메시지로 **데이터 경계**를 인지시킬 수 있음.

---

### 8.4 ML 관점

* 전체 데이터 분포가 바뀌는 이벤트(새 도메인, 새 언어 등)가 발생했는지
  공통 레벨에서 감지하는 것도 중요.
* 예: HR 폴더에 새로운 언어(일본어) 문서가 대량 추가되면:

  * 임베딩/LLM이 그 언어에서 제대로 동작하는지 점검 필요.

---

* 서버리스 + 무료 모델 STT/LLM/임베딩 조합
* `scan → train → chat → watch` 파이프라인
* Meeting / Knowledge / 추가 에이전트의 공통 Cycle 구조

  “Cycle 0~5 중 어디에 영향을 주는지”를 판단하는 기준으로 활용할 수 있다.
